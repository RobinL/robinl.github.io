# Optimising DuckDB performance on large EC2 instances

I have been [recommending for some time](/recommend_duckdb/) that DuckDB is often a faster and simpler choice than Spark for [most data engineering workloads](https://www.fivetran.com/blog/how-do-people-use-snowflake-and-redshift).[^1]

[^1]: Hat tip to [this](https://youtu.be/DxwDaoUijTc?si=Pkmw0GwxfPOn10ea&t=934) presentation where I first saw this chart.

For big data workloads, this suggests processing data on a single very [large EC2 instance](https://instances.vantage.sh/aws/ec2/r8id.96xlarge?currency=USD) with hundreds of vCPUs.

**However, there are some performance footguns with this approach**: the default configuration can result in surprisingly poor performance, and a few simple changes can make workloads run multiple times faster.

So how can we optimise performance?  This post sets out results from a variety of benchmarking experiments (code [here](https://github.com/RobinL/duckdb_large_ec2_benchmarking)).

## Key findings

#### High thread counts and performance
- [Do not assume](https://duckdb.org/docs/current/guides/performance/how_to_tune_workloads#too-many-threads) more threads always gives better performance.  On some workloads performance doubled or more when using fewer threads, but is dependent on workload and machine type, so experiment with different settings.
- Do not assume that 100% usage of all cores means they're being used efficiently: the workload may still run faster on fewer threads.
- Running multiple thread-constrained queries in parallel does not save time vs. running them serially, since the queries run slower when run in parallel.

#### Memory Bandwidth:
- Memory bandwith on large EC2 instances was mesured at about 330GB/s (`r8id.32xlarge`) to 480GB/s (`r8id.metal-48xl`). Memory bandwith on an Macbook Pro M4 Max is 410-546GB/s, so for memory bandwidth constrained workloads, we may not expect the larger CPU counts on EC2 to give much of a performance boost over a high-end laptop.
- However, for the workloads tested, I was not able to find evidence they were bottlenecked by memory bandwidth.

#### Disk IO:
- Throughput of writing to disk can be an important bottleneck.  Use [instance store](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ssd-instance-store.html) (SSDs), or if unavailable, set [EBS throughput](https://docs.aws.amazon.com/ebs/latest/userguide/ebs-volume-types.html) to maximum.


#### S3 Transfer
- With the [CRT transfer client](https://aws.amazon.com/blogs/developer/introducing-crt-based-s3-client-and-the-s3-transfer-manager-in-the-aws-sdk-for-java-2-x/) enabled, `aws s3 sync` can be substantially faster than DuckDB's built-in S3 write capabilities.  Therefore DuckDB -> local SSD -> S3 via `aws s3 sync` can be faster than writing to S3 directly from DuckDB.


## Evidence

What follows contains more detailed evidence to substantiate the above findings.

All results used `duckdb==1.5.2` and were run on instances in the [`r8id` family](https://instances.vantage.sh/aws/ec2/r8id.32xlarge), mostly `r8id.32xlarge` (128 vCPU, 1 TiB RAM, 2x3.8TB NVMe SSD in RAID 0).

### High thread counts and performance

I ran the following simple query against 800m rows of data, first on an `r8id.32xlarge` and then, for the raw Parquet path, on an `r8id.metal-48xl`.

```sql
 SELECT col_1, col_2, COUNT(*) AS cnt
 FROM benchmark_input
 GROUP BY col_1, col_2
 ORDER BY cnt DESC
 LIMIT 10
```

The performance varied depending on the DuckDB [`THREADS` pragma setting](https://duckdb.org/docs/current/configuration/pragmas.html#threads):

### Does running multiple queries in parallel help?

If fewer threads can improve performance, a natural follow up is to test whether that means we can run multiple queries in parallel, each with fewer threads.

To test this, I ran the group/order query against raw Parquet files on the `r8id.32xlarge`, but launched multiple independent queries in parallel.

<VegaLite client:visible spec={parallel_query_runtime_chart} />

Further experiments suggested it's also possible for it to make things worse by running in parallel, e.g. that running four 32 thread queries serially was faster than running them in parallel.

Overall this evidence suggests suggesting the machine is bottlenecked when running at lower thread counts, just not by CPU, and I would not recommend trying to run queries in parallel.


### Memory bandwith

I conducted various experiments to see whether these bottlenecks could be explained by memory bandwidth.  I couldn't find any conclusive evidence to suggest memory bandwidth was the bottleneck.


<details>
<summary>Click to expand memory bandwidth analysis</summary>
One possible explanation would be that the machine is simply running out of memory bandwidth.



<VegaLite client:visible spec={memory_bandwidth_chart} />

This makes a simple memory-bandwidth explanation look unlikely. The synthetic triad benchmark gets about 153 GiB/s at 32 threads and about 309 GiB/s at 64 or 128 threads. If the DuckDB query were saturating main memory bandwidth at 32 threads, I would expect the 64-thread single-query run, or the two-query 64-thread run, to improve materially. They do not.

The phase-level machine metrics point the same way. During the two-query and four-query runs, average host CPU usage was only around 14-23%, and the monitor recorded almost no disk I/O during the query phases, probably because the Parquet data was already hot in the OS page cache.

My best hypothesis is therefore that the bottleneck is not raw hardware bandwidth, but some combination of DuckDB execution overhead, synchronisation, cache locality, and stages of the scan/group/order pipeline that do not scale smoothly when more queries are competing at once. The practical conclusion is that parallel sessions are not a reliable way to use the spare cores on this workload: benchmark it, and prefer a smaller number of concurrent queries unless measured throughput actually improves.

</details>

### Disk I/O

DuckDB can be significantly bottlenecked by disk throughput when writing large files.

My benchmarks showed it was capable of writing at about 8GiB/s to the two SSDs (connected in RAID0) on the `r8id.32xlarge`, similar to their expected maximum performance.

The benchmarks also showed that slower disk will cause correspondingly slower runtimes as you wait for data to be written to disk.  EBS throughput is configurable, but gp3 can default to just 125 MiB/s, so the slowdown can be dramatic.

Since DuckDB will often [spill to disk](https://duckdb.org/2024/07/09/memory-management#intermediate-spilling) this finding is relevant even for operations that do not explicitly write out large datasets to disk.

Write performance is tested by converting `.parquet` files into a 4.9GiB `mydb.duckdb` database file on disk using a `CREATE TABLE AS SELECT *` statement.

<VegaLite client:visible spec={disk_io_chart} />

On EBS, we can see the long 'tail' of low CPU activity whilst the machine is waiting on disk writes to complete. We can be more precise about this by running the benchmark with `RECOVERY_MODE=no_wal_writes` (see [here](https://duckdb.org/docs/current/sql/statements/attach#:~:text=writes%20disabled%20for%20improved%20performance)), which splits the workload into a `CTAS` phase and a `CHECKPOINT` phase, in which the data is written. This is shown in the lower charts.

For this run, EBS throughput was set to 1000 MiB/s. We can see the `CHECKPOINT` phase takes around 4.1 seconds, consistent with the final 4.9 GiB database. Further experiments with different throughput showed the length of the `CHECKPOINT` phase scales proportionately with EBS throughput.

EBS throughput can be increased to a [maximum of 2,000 MiB/s](https://docs.aws.amazon.com/ebs/latest/userguide/general-purpose.html#gp3-performance), but a better option is to use instance storage (SSDs), which can achieve much higher effective DuckDB write rates.

### S3 I/O

#### Writes
For S3 writes, I benchmarked an 800 million-row table on an `r8id.32xlarge` with DuckDB `threads=64` in two ways:

1. A direct DuckDB `COPY` to S3,
2. DuckDB writing Parquet to local NVMe SSD before uploading with `aws s3 sync`.

Further research showed `aws s3 sync` [is expected to be slow](https://docs.aws.amazon.com/cli/latest/topic/s3-config.html#max-concurrent-requests) with its default classic transfer client settings, and a [recommended solution is to use the CRT transfer client instead](https://aws.amazon.com/blogs/storage/improving-amazon-s3-throughput-for-the-aws-cli-and-boto3-with-the-aws-common-runtime/).

The output was 8.3 GiB across 64 Parquet files.

Results are shown below. Hover over a segment to see the stage throughput and the overall throughput for that option.

<VegaLite client:visible spec={s3_io_chart} />

The DuckDB -> S3 direct route doesn't come close to saturating S3 bandwidth, at 998MiB/s.  `s3 sync` with CRT is much faster at 2,572 MiB/s, closer to the quoted EC2 network performance of [50 Gbit/s, or about 5,960 MiB/s](https://docs.aws.amazon.com/ec2/latest/instancetypes/mo.html#mo_network).

I experimented with DuckDB's [s3 settings](https://duckdb.org/docs/current/configuration/overview#:~:text=S3%20Uploader%20global%20thread%20limit) for the DuckDB-to-S3 write, but they did not make a significant difference.

#### Reads

I used the same `aws s3 sync` approach to download the benchmark dataset from S3 to local SSD. With default settings, downloading the 3.9 GiB dataset took about 8 seconds, which is about 460.0 MiB/s.

After enabling the CRT transfer client for the worker-side dataset sync, the same download completed in about 1 second, implying a throughput of about 3,680.0 MiB/s. That is roughly an 8x improvement.

As with writes, this still does not quite saturate the quoted EC2 network bandwidth for this instance type of [about 5,960 MiB/s](https://docs.aws.amazon.com/ec2/latest/instancetypes/mo.html#mo_network), but it is much closer than the default transfer path.
