# Optimising DuckDB performance on large EC2 instances

As DuckDB has matured, we now [recommend it](https://moj-analytical-services.github.io/splink/topic_guides/splink_fundamentals/backends/backends.html?h=recommend#duckdb) over Spark for most record linkage workloads using Splink.

But how can we get optimal performance out of DuckDB on EC2 instances?  This post sets out results from a variety of benchmarking experiments.

## Key findings

- [Do not assume](https://duckdb.org/docs/current/guides/performance/how_to_tune_workloads#too-many-threads) more threads always gives better performance.  On some workloads performance doubled or more when using fewer threads.
- Throughput of writing to disk can be an important bottleneck.  Use [instance store](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ssd-instance-store.html) (SSDs), or if unavailable, set EBS thoughput to maximum.
- DuckDB is relatively slow when writing to S3, and so is `aws s3 sync` under default settings.  For optimsal write speed, write from DuckDB to local SSD and then to S3 using the [CRT transfer client](https://aws.amazon.com/blogs/developer/introducing-crt-based-s3-client-and-the-s3-transfer-manager-in-the-aws-sdk-for-java-2-x/).

## Evidence

```sql
 SELECT col_1, col_2, COUNT(*) AS cnt
 FROM benchmark_input
 GROUP BY col_1, col_2
 ORDER BY cnt DESC
 LIMIT 10
```

### Disk I/O

My experiments show that DuckDB is capable of persisting a `.duckdb` file to local RAID0 instance storage at about 8.4 GiB/s on an `r8id.32xlarge`, but the default throughput of EBS [is just 125 MiB/s](https://docs.aws.amazon.com/ebs/latest/userguide/general-purpose.html#gp3-performance), so if you write large amounts of data to EBS under default settings, runtime can be dominated by the disk bottleneck.

EBS throughput can be increased to a [maximum of 2,000 MiB/s](https://docs.aws.amazon.com/ebs/latest/userguide/general-purpose.html#gp3-performance), but a better option is to use instance storage (SSDs), which can achieve much higher effective DuckDB write rates.

For local disk writes, the important question is whether the bottleneck is DuckDB's CPU work, or the final persistence of the `.duckdb` file to storage. A useful diagnostic here was running the create-db benchmark with `RECOVERY_MODE=no_wal_writes`, which splits the workload into a `CTAS` phase and a later `CHECKPOINT` phase. The resulting database file was about 4.9 GiB, so the implied throughputs below should be read as effective DuckDB persistence rates rather than raw device benchmarks.

The chart below resets each panel to time zero and plots CPU as a percentage of the requested 64 DuckDB threads. In the `split` panels, the line runs straight from `CTAS` into `CHECKPOINT`, so the important comparison is the shape and total duration of the run rather than wall-clock alignment across experiments.

<VegaLite client:visible spec={disk_io_chart} />

The pattern was very consistent. On the `r8id.8xlarge` with 32 threads, `parquet -> duckdb` took 10.32s of `CTAS` on EBS and 10.25s on SSD, but `CHECKPOINT` took 9.07s on EBS and 1.96s on SSD. On the `r8id.32xlarge` with 64 threads, `CTAS` was again almost unchanged, at 5.86s on EBS and 5.65s on SSD, while `CHECKPOINT` was 9.09s on EBS and 0.89s on SSD. The same pattern appeared for `duckdb -> duckdb`, where the 64-thread `CHECKPOINT` times were 9.09s on EBS and 0.88s on SSD. That strongly suggests the storage-sensitive part is persistence/writeback, not Parquet decoding or DuckDB materialisation.

It also shows that simply moving to a larger instance does not fix a storage bottleneck. The `r8id.8xlarge` and `r8id.32xlarge` runs both spent about 9 seconds in `CHECKPOINT` on the lower-throughput EBS configuration, even though the larger machine cut the `CTAS` phase roughly in half. After increasing EBS throughput on the `r8id.32xlarge`, the 64-thread EBS `CHECKPOINT` fell to 4.12s. End-to-end runtime moved with it: `parquet -> duckdb` on EBS dropped from 15.23s to 10.30s, and `duckdb -> duckdb` dropped from 14.93s to 9.81s. That is strong evidence that EBS throughput, not CPU, was the limiting factor.

Local instance storage was still much faster. On the `r8id.32xlarge`, the single NVMe drive checkpointed the ~4.9 GiB database in about 0.9s, while RAID0 across the two local NVMe drives cut that further to about 0.6s. That corresponds to an effective persistence rate of roughly 5.5 GiB/s for the single-drive setup and about 8.4 GiB/s for RAID0, versus roughly 0.5 GiB/s for the lower-throughput EBS configuration. The practical conclusion is that if a workload writes a lot of DuckDB data, provisioned EBS throughput matters a lot, but local instance-store SSD is the better hot working disk when it is available.





### S3 IO

#### Writes
For S3 writes, I benchmarked the a 800 million-row table on an `r8id.32xlarge` with DuckDB `threads=64` in two ways:

1. A direct DuckDB `COPY` to S3,
2. DuckDB writing Parquet to local NVMe SSD before uploading with `aws s3 sync`.

Further research showed `aws s3 sync` [is expected to be slow](https://docs.aws.amazon.com/cli/latest/topic/s3-config.html#max-concurrent-requests) with its default classic transfer client settings, and a [recommended solution is to use the CRT transfer client instead](https://aws.amazon.com/blogs/storage/improving-amazon-s3-throughput-for-the-aws-cli-and-boto3-with-the-aws-common-runtime/).

The output was 8.3 GiB across 64 Parquet files.

Results are shown below. Hover over a segment to see the stage throughput and the overall throughput for that option.

<VegaLite client:visible spec={s3_io_chart} />

The DuckDB -> S3 direct route doesn't come close to saturating S3 bandwidth, at 998MiB/s.  `s3 sync` with CRT is much faster at 2,572 MiB/s, but still falls short of the quoted EC2 network bandwidth for this instance type of [50 Gbit/s, or about 5,960 MiB/s](https://docs.aws.amazon.com/ec2/latest/instancetypes/mo.html#mo_network).


I experimented with DuckDB's [s3 settings](https://duckdb.org/docs/current/configuration/overview#:~:text=S3%20Uploader%20global%20thread%20limit) for the DuckDB-to-S3, but they did not make a significant difference.

#### Reads

I used the same `aws s3 sync` approach to download the benchmark dataset from S3 to local SSD. With default settings, downloading the 3.9 GiB dataset took about 8 seconds, which is about 460.0 MiB/s.

After enabling the CRT transfer client for the worker-side dataset sync, the same download completed in about 1 second, implying a throughput of about 3,680.0 MiB/s. That is roughly an 8x improvement.

As with writes, this still does not quite saturate the quoted EC2 network bandwidth for this instance type of [about 5,960 MiB/s](https://docs.aws.amazon.com/ec2/latest/instancetypes/mo.html#mo_network), but it is much closer than the default transfer path.
